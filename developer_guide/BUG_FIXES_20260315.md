## 1. Logout Button Not Showing When Logged In

**File:** `resources/js/Components/Jetstream/PublicDigitHeader.vue`

**Symptom:** Even when authenticated and on the dashboard, the header showed the Login button instead of the Logout button.

**Root Cause:**
```javascript
// WRONG — auth.user does not exist in this app's shared props
const isLoggedIn = computed(() => !!page.props.auth?.user)
```
`HandleInertiaRequests.php` shares the user at the **top level** (`page.props.user`), not nested under `auth`. The check always returned `false`.

**Fix:**
```javascript
// CORRECT — matches what HandleInertiaRequests actually shares
const isLoggedIn = computed(() => !!page.props.user)
```

**Rule:** Always check `HandleInertiaRequests::share()` to know the exact shape of shared props before writing `page.props.*` checks.

## 2. Logout Function Broken (Inertia 2.0)

**File:** `resources/js/Components/Jetstream/PublicDigitHeader.vue`

**Symptom:** Clicking Logout did nothing or returned a 302 HTML redirect instead of handling the response correctly.

**Root Cause:** The component used `useForm` + `logoutForm.post()` — the Inertia 1.0 API. This is broken in Inertia 2.0 in Composition API context.

```javascript
// WRONG — Inertia 1.0 style, broken in Inertia 2.0 Composition API
const logoutForm = useForm({})
logoutForm.post(route('logout'), { ... })
```

**Fix:**
```javascript
// CORRECT — Inertia 2.0 Composition API
import { router } from '@inertiajs/vue3'

const isLoggingOut = ref(false)

function logout() {
    isLoggingOut.value = true
    router.post(route('logout'), {}, {
        preserveState: false,
        onFinish: () => { isLoggingOut.value = false },
        onError: () => { isLoggingOut.value = false }
    })
}
```

**Rule:** In Composition API components, always use `router.post()` from `@inertiajs/vue3` for form submissions. `useForm` works but `logoutForm.post()` requires the Options API `$inertia` context to behave correctly with redirects.

**Note:** `ElectionNavigation.vue` uses Options API (`this.$inertia.post()`) which still works fine — this issue only affected the Composition API header.

## 3. Vue I18n Nesting Errors

**Files:**
- `resources/js/Pages/Vote/DemoVote/Create.vue`
- `resources/js/locales/pages/Vote/DemoVote/Create/en.json`
- `resources/js/locales/pages/Vote/DemoVote/Create/de.json`
- `resources/js/locales/pages/Vote/DemoVote/Create/np.json`

**Symptom:** Console errors on the voting page:
```
Message compilation error: Not allowed nest placeholder
1  |  Welcome {{ user_name }}!
Message compilation error: Not allowed nest placeholder
1  |  Candidate for {{ post_name }}
```

**Root Cause:** Vue I18n v9+ does not allow `{{ }}` Mustache syntax inside translation strings. The locale JSON files contained:
```json
"welcome": "Welcome {{ user_name }}!",
"candidates_label": "Candidate for {{ post_name }}"
```
These were being passed as named parameters via `$t('key', { user_name: ... })` but Vue I18n v9 uses `{param}` (single braces) for interpolation, not `{{ }}`.

**Fix — Locale JSON files:** Remove the placeholders entirely, use plain text:
```json
"welcome": "Welcome",
"candidates_label": "Candidate for"
```

**Fix — Vue template:** Append the dynamic value directly in the template:
```vue
<!-- BEFORE (broken) -->
{{ $t('...welcome', { user_name: user_name }) }}

<!-- AFTER (correct) -->
{{ $t('...welcome') }} {{ user_name }}!
```
```vue
<!-- BEFORE (broken) -->
{{ $t('...candidates_label', { post_name: post.name }) }}

<!-- AFTER (correct) -->
{{ $t('...candidates_label') }} {{ post.name }}
```

**Rule:** In Vue I18n v9, named interpolation uses `{param}` (single braces). If the translation string already has `{{ }}`, either fix the JSON to use `{param}` syntax, or remove the placeholder from the JSON and concatenate in the template.

## 4. Candidate Images Returning 404

**Files:**
- `app/Services/DemoElectionCreationService.php`
- `resources/js/Pages/Vote/DemoVote/Create.vue`
- Database: `demo_candidacies.image_path_1`

**Symptom:**
```
GET /storage/candidates/alice-johnson_president_01.png 404
GET /storage/candidates/bob-smith_president_02.png 404
```

**Root Cause — Service:** `getCandidateImagePath()` was generating name-based slugged paths:
```php
return "candidates/{$nameSlug}_{$postSlug}{$regionSlug}_01.png";
// → candidates/alice-johnson_president_01.png  (file never existed)
```

**Root Cause — DB records:** The `DemoCandidacy` model has a `BelongsToTenant` global scope. When `demo:setup --force` was run, the scope hid existing records so a new election was created instead of updating the old one's candidacies. The old records kept the broken paths.

**Root Cause — Vue:** `getImageUrl()` was prepending `/storage/` to the already-broken path, serving a 404 URL.

**Fix — Service:** Changed `getCandidateImagePath()` to cycle through the 8 available demo images:
```php
private function getCandidateImagePath(string $name, string $postName, ?string $region, int $index): string
{
    $imageNumber = ($index % 8) + 1;
    return "demo_candiate_images/candidates_" . str_pad($imageNumber, 2, '0', STR_PAD_LEFT) . ".png";
}
```

**Fix — DB records (raw query to bypass global scope):**
```php
// Must use DB::table() directly — DemoCandidacy model has BelongsToTenant scope
// which hides records when no organisation session context is set
$rows = DB::table('demo_candidacies')
    ->where('image_path_1', 'like', 'candidates/%')
    ->orderBy('position_order')
    ->get(['id']);

foreach ($rows as $i => $row) {
    $imageNumber = ($i % 8) + 1;
    $path = 'demo_candiate_images/candidates_' . str_pad($imageNumber, 2, '0', STR_PAD_LEFT) . '.png';
    DB::table('demo_candidacies')->where('id', $row->id)->update(['image_path_1' => $path]);
}
```

**Fix — Vue:** Simplified `getImageUrl()` since paths are now correct at the source:
```javascript
const getImageUrl = (path) => {
    if (!path) return null
    if (path.startsWith('http') || path.startsWith('/storage')) return path
    return `/storage/${path}`
}
```

**Image files location:** `public/storage/demo_candiate_images/candidates_01.png` through `candidates_08.png`

**Rule:** When running artisan commands that bypass Eloquent global scopes, use `DB::table()` directly to verify data. Always store image paths relative to the `storage/app/public/` directory.

## 5. Candidate Names Had Taglines

**File:** `app/Services/DemoElectionCreationService.php`

**Symptom:** Voting page showed `Alice Johnson - Progressive Platform` instead of just `Alice Johnson`.

**Root Cause:** The service defined `candidacy_name` with appended taglines:
```php
['name' => 'Alice Johnson', 'candidacy_name' => 'Alice Johnson - Progressive Platform'],
```

**Fix — Service:** Set `candidacy_name` equal to just the name:
```php
['name' => 'Alice Johnson', 'candidacy_name' => 'Alice Johnson'],
```

**Fix — DB records:** Updated all existing records directly:
```php
$updates = [
    'Alice Johnson - Progressive Platform' => 'Alice Johnson',
    'Bob Smith - Economic Growth'          => 'Bob Smith',
    // ... all 11 candidates
];
foreach ($updates as $old => $new) {
    DB::table('demo_candidacies')->where('candidacy_name', $old)->update(['candidacy_name' => $new]);
}
```

## 6. Emails Stopped Sending

**File:** `C:/php-8.3.24/php.ini`

**Symptom:** All outgoing emails silently failed. Laravel log showed:
```
Connection could not be established with host "ssl://smtp.hostinger.com:465":
failed loading cafile stream: `cacert.pem'
```

**Root Cause:** `php.ini` had `cacert.pem` as a **relative path**:
```ini
curl.cainfo = "cacert.pem"
openssl.cafile = "cacert.pem"
```
PHP resolves relative paths from its working directory at runtime. After a PHP update to `php-8.3.24`, the working directory context changed and PHP could no longer locate the file, even though `cacert.pem` physically existed at `C:/php-8.3.24/cacert.pem`.

**Fix:** Changed to absolute paths in `php.ini`:
```ini
curl.cainfo = "C:\php-8.3.24\cacert.pem"
openssl.cafile = "C:\php-8.3.24\cacert.pem"
```

**Verification:**
```php
Mail::raw('Test', fn($m) => $m->to('test@example.com')->subject('Test'));
// Output: Mail sent successfully
```

**Rule:** On Windows, always use absolute paths for `curl.cainfo` and `openssl.cafile` in `php.ini`. Relative paths are fragile and break after PHP version upgrades or working directory changes. The `cacert.pem` file can be downloaded from `https://curl.se/ca/cacert.pem` if missing.
