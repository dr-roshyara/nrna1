# Bug Fixes — 2026-03-14

Two unrelated bugs were fixed in the same session. This guide explains what was wrong,
why it was wrong, how to detect it, and how it was fixed — so it never has to be
debugged from scratch again.

---

## Bug 1: Vue i18n "Invalid linked format" SyntaxError

### Symptom

The browser console showed this error whenever the **Organisation Create Modal**
was opened from `/dashboard/welcome`:

```
SyntaxError: Invalid linked format (at vendor-BuPF5z_P.js:113:2267)
    at rs (vendor-BuPF5z_P.js:113:2267)
    at zC (vendor-BuPF5z_P.js:115:518)
```

The modal refused to render. The error appeared to come from the bundled vendor
library, making it hard to trace back to source code.

---

### Root Cause

**vue-i18n v9 reserves the `@` character.** It uses `@:key` syntax for "linked
messages" (re-using one translation string inside another). If vue-i18n's message
compiler encounters a bare `@` that is NOT followed by `:key`, it throws:

```
SyntaxError: Invalid linked format
```

Translation strings that contained email addresses — either as literal `@` or as
the Unicode escape `\u0040` (which JSON decodes to `@` at runtime) — were being
passed through vue-i18n's message compiler and triggering the error.

The problem appeared in **three locations**:

#### 1. Locale JSON files with `\u0040` (Unicode-escaped `@`)

```
resources/js/locales/pages/Dashboard/organisation/de.json
resources/js/locales/pages/Dashboard/organisation/en.json
resources/js/locales/pages/Dashboard/organisation/np.json
resources/js/locales/pages/Auth/ForgotPassword/de.json
resources/js/locales/pages/Auth/ForgotPassword/en.json
resources/js/locales/pages/Auth/ForgotPassword/np.json
resources/js/locales/pages/faq/de.json
```

Example of the problematic value:

```json
"email_placeholder": "vorstand\u0040tv-muenchen.de"
```

`\u0040` decodes to `@` — so vue-i18n saw `vorstand@tv-muenchen.de` and tried to
parse `@tv-muenchen.de` as a linked message reference.

#### 2. Inline `fallback` strings in `$t()` calls inside Vue components

```
resources/js/Components/Organisation/Steps/OrganizationStepBasicInfo.vue
resources/js/Components/Organisation/Steps/OrganizationStepRepresentative.vue
```

Example:

```vue
:placeholder="$t('organisation.form.email_placeholder', { fallback: 'vorstand@tv-muenchen.de' })"
```

The `fallback` key is treated by vue-i18n as a named interpolation value
(`{fallback}` placeholder substitution). When the translation key resolves to a
string that contains `{fallback}`, vue-i18n substitutes in the fallback value and
then compiles the result — hitting the `@` in the email address.

---

### Why It Was Hard to Find

- The error stack trace points to the minified vendor bundle (`vendor-BuPF5z_P.js`),
  not to source files.
- `\u0040` in JSON is invisible unless you read the raw file bytes — grep for `@`
  finds nothing because `\u0040` is not the `@` byte.
- The error only fires when the modal opens (lazy compilation), not on page load.

---

### How to Detect It in Future

Run this command before every build to catch `@` in locale files:

```bash
# Find literal @ in locale files
grep -rn "@" resources/js/locales/

# Find Unicode-escaped @ (\u0040) in locale files
grep -rn "u0040" resources/js/locales/

# Find @ inside $t() fallback strings in Vue/JS files
grep -rn "fallback:.*@" resources/js/ --include="*.vue" --include="*.js"
```

All three should return zero results.

---

### The Fix

**Rule: Never put an `@` character (or `\u0040`) inside any vue-i18n message string.**

Replace email-style placeholders with descriptive text:

| File | Before | After |
|------|--------|-------|
| `Dashboard/organisation/de.json` | `"vorstand\u0040tv-muenchen.de"` | `"E-Mail-Adresse"` |
| `Dashboard/organisation/en.json` | `"board\u0040gym-munich.de"` | `"Email address"` |
| `Dashboard/organisation/np.json` | `"board\u0040organization.np"` | `"इमेल ठेगाना"` |
| `Dashboard/organisation/de.json` | `"t.schmidt\u0040tv-muenchen.de"` | `"E-Mail-Adresse"` |
| `Dashboard/organisation/en.json` | `"t.schmidt\u0040gym-munich.de"` | `"Email address"` |
| `Dashboard/organisation/np.json` | `"t.schmidt\u0040organization.np"` | `"इमेल ठेगाना"` |
| `ForgotPassword/de.json` | `"ihre@email.de"` | `"E-Mail-Adresse eingeben"` |
| `ForgotPassword/en.json` | `"your@email.com"` | `"Enter your email address"` |
| `ForgotPassword/np.json` | `"your@email.com"` | `"इमेल ठेगाना प्रविष्ट गर्नुहोस्"` |
| `faq/de.json` | `"support\u0040publicdigit.com"` | `"support(at)publicdigit.com"` |
| `OrganizationStepBasicInfo.vue` | `fallback: 'vorstand@tv-muenchen.de'` | `fallback: 'E-Mail-Adresse'` |
| `OrganizationStepRepresentative.vue` | `fallback: 't.schmidt@tv-muenchen.de'` | `fallback: 'E-Mail-Adresse'` |

The i18n configuration in `resources/js/i18n.js` was also hardened:

```js
const i18n = createI18n({
  legacy: false,
  locale: initialLocale,
  fallbackLocale: 'en',
  messages,
  globalInjection: true,
  missingWarn: false,
  fallbackWarn: false,
  warnHtmlMessage: false,   // added
  onWarn: () => {},          // added — suppresses remaining vue-i18n warnings
});
```

---

### Rebuild Required After Locale File Changes

Locale files are bundled at build time. After any change to a `.json` locale file
or a `.vue` component, rebuild:

```bash
npm run build
```

## Bug 2: 403 on `/v/{vslug}/demo-code/create` in Production

### Symptom

Clicking the "Start Demo" button in production redirected through the slug-based
voting flow and produced a **403 Forbidden** response when Inertia navigated to:

```
/v/{vslug}/demo-code/create
```

It worked fine locally. The browser showed the 403 as an Inertia navigation error.

---

### Root Cause

`EnsureVoterStepOrder` middleware read the `vslug` route parameter using
`$request->route('vslug')`. This **always returns the raw string** (the slug hash),
because the `{vslug}` route parameter has no Eloquent route model binding
configured.

```php
// WRONG — returns the raw string "tbjThp50VKxMjoFUxKojkn4cxlmbXSTqH"
$vslug = $request->route('vslug');

// A string is never instanceof VoterSlug or DemoVoterSlug
if (!$vslug instanceof VoterSlug && !$vslug instanceof DemoVoterSlug) {
    abort(403, 'Invalid voting link.');   // ← ALWAYS fired
}
```

The middleware chain already had `voter.slug.verify` (`VerifyVoterSlug`) running
**before** `voter.step.order` (`EnsureVoterStepOrder`). `VerifyVoterSlug` correctly
looks up the model and stores it:

```php
// In VerifyVoterSlug — correct, runs first
$request->attributes->set('voter_slug', $voterSlug);
```

But `EnsureVoterStepOrder` was not reading from `$request->attributes` — it was
re-reading the raw route string, which is never a model.

---

### Why It Appeared to Work Locally

Local testing used the non-slug route:

```
/demo/code/create        ← no middleware EnsureVoterStepOrder
```

Production uses the slug route:

```
/v/{vslug}/demo-code/create   ← has EnsureVoterStepOrder in chain
```

So the bug existed everywhere but was only triggered by the slug-based flow used in
production.

---

### The Middleware Chain (slug routes)

```
auth:sanctum
verified
SubstituteBindings
voter.slug.verify       ← resolves & validates slug, stores model in $request->attributes
voter.slug.window       ← checks expiry
voter.slug.consistency  ← checks election/org match
voting.code.window      ← checks voting window
voter.step.order        ← BUG WAS HERE: read route string instead of attributes model
vote.eligibility
vote.organisation
election.demo
```

Data flows **through request attributes**, not through re-reading route parameters.
Any middleware after `voter.slug.verify` must use `$request->attributes->get('voter_slug')`.

---

### The Fix

**File:** `app/Http/Middleware/EnsureVoterStepOrder.php`

```php
// BEFORE (broken)
$vslug = $request->route('vslug');

// AFTER (correct)
$vslug = $request->attributes->get('voter_slug');
```

One line change. The rest of the middleware logic was correct.

---

### Rule for Future Middleware in This Chain

Any middleware in the `v/{vslug}/*` route group that needs the voter slug model
**must** read it from request attributes:

```php
// Always use this in slug-route middleware
$vslug = $request->attributes->get('voter_slug');

// Never use this to get the model (gives you a string)
$vslug = $request->route('vslug');
```

The same rule applies to the election model, which `VerifyVoterSlugConsistency`
sets after its own checks:

```php
// Set by VerifyVoterSlugConsistency
$request->attributes->set('election', $election);

// Read in later middleware/controllers like this
$election = $request->attributes->get('election');
```
